# Anlyx v0.1 Release Runbook

## Purpose

This runbook describes the manual steps required immediately before and after publishing Anlyx v0.1 packages to npm.

This document is not a publish log. PR 21 does not run `npm publish`, create Git tags, create GitHub Releases, change package names, or add release automation.

## Packages

Anlyx v0.1 is published as these npm packages:

- `@anlyx/core`
- `@anlyx/ui`
- `@anlyx/adapter-openapi`
- `@anlyx/adapter-manual`
- `@anlyx/adapter-next`
- `@anlyx/adapter-spring`
- `@anlyx/capture`
- `anlyx`

Do not rename packages during the release. If a registry or scope problem is found, stop and use the migration plan in this runbook.

## 1. Prerequisites

Run these checks from the repository root before publishing.

```bash
npm whoami
npm profile get tfa
git remote -v
git status
```

Required results:

- `npm whoami` returns the npm account that should own the Anlyx packages.
- npm 2FA status is known before publish starts.
- If publishing with a token, the token has publish permission for unscoped and scoped packages.
- GitHub `origin` points to `https://github.com/suhannoh/anlyx.git` or `git@github.com:suhannoh/anlyx.git`.
- The working tree is clean.

If any prerequisite fails, stop the release. Do not publish from an unknown account, dirty tree, or unexpected remote.

## 2. npm Package Name and Scope Checks

Check package name availability and current registry ownership before publishing.

```bash
npm view anlyx name
npm view @anlyx/core name
npm view @anlyx/ui name
npm view @anlyx/adapter-spring name
npm view @anlyx/adapter-next name
npm view @anlyx/adapter-openapi name
npm view @anlyx/adapter-manual name
npm view @anlyx/capture name
```

Interpretation:

- If a command returns a package name, that package may already exist. Confirm ownership and version before publishing.
- If npm returns `404`, the package likely does not exist yet.
- Publishing `@anlyx/*` requires npm permission for the `@anlyx` scope or organization.
- If the `@anlyx` scope cannot be used, do not publish under a different scope ad hoc.

### `@anlyx` Scope Fallback Plan

If `@anlyx` scope permission is denied and cannot be resolved:

1. Stop the release before publishing any package.
2. Open a follow-up PR that changes internal package names from `@anlyx/*` to `@suhannoh/*`.
3. Update all workspace dependencies, package exports, README usage notes, release docs, and any import paths.
4. Run the full verification and pack dry-run again.
5. Publish only after the new package naming is reviewed.

The CLI package name `anlyx` should remain unchanged unless the unscoped name is unavailable or owned by someone else.

## 3. Final Verification

Run the full local verification suite before publishing.

```bash
corepack pnpm install
corepack pnpm typecheck
corepack pnpm lint
corepack pnpm test
corepack pnpm format
corepack pnpm -r build
git diff --check
git status
```

Required results:

- All commands exit successfully.
- `git status` is clean.
- No generated or temporary tarballs are staged.

## 4. Pack Dry-Run

Run package dry-runs in publish order.

```bash
corepack pnpm --filter @anlyx/core pack --dry-run
corepack pnpm --filter @anlyx/ui pack --dry-run
corepack pnpm --filter @anlyx/adapter-openapi pack --dry-run
corepack pnpm --filter @anlyx/adapter-manual pack --dry-run
corepack pnpm --filter @anlyx/adapter-next pack --dry-run
corepack pnpm --filter @anlyx/adapter-spring pack --dry-run
corepack pnpm --filter @anlyx/capture pack --dry-run
corepack pnpm --filter anlyx pack --dry-run
```

Check every tarball summary:

- `dist` files are present.
- `README.md`, `LICENSE`, and `package.json` are present.
- Source tests, fixture data, and unrelated repository docs are not included.
- The `anlyx` package includes `dist/index.js` and the `bin` entry points to it.

## 5. Publish Order

Publish internal dependencies before dependents.

1. `@anlyx/core`
2. `@anlyx/ui`
3. `@anlyx/adapter-openapi`
4. `@anlyx/adapter-manual`
5. `@anlyx/adapter-next`
6. `@anlyx/adapter-spring`
7. `@anlyx/capture`
8. `anlyx`

Scoped packages need public access on first publish.

```bash
cd packages/core
npm publish --access public
```

Repeat for each scoped package in order:

```bash
cd packages/ui
npm publish --access public

cd ../adapter-openapi
npm publish --access public

cd ../adapter-manual
npm publish --access public

cd ../adapter-next
npm publish --access public

cd ../adapter-spring
npm publish --access public

cd ../capture
npm publish --access public
```

Publish the unscoped CLI package with normal npm publish.

```bash
cd ../cli
npm publish
```

Do not run these commands during PR review. They are for the approved release moment only.

## 6. Post-Publish Verification

After publishing, verify registry state and a fresh install.

```bash
npm view anlyx version
npm view @anlyx/core version
```

Then test installation in a temporary directory.

```bash
mkdir /tmp/anlyx-install-test
cd /tmp/anlyx-install-test
npm init -y
npm install anlyx
npx anlyx --help
npx anlyx init --force
```

Expected results:

- `npm view` returns `0.1.0` for published packages.
- `npm install anlyx` completes without unresolved `@anlyx/*` dependencies.
- `npx anlyx --help` prints the CLI usage.
- `npx anlyx init --force` creates `anlyx.config.ts`.

## 7. Failure Response

### Package Name Already Taken

- Stop before publishing additional packages.
- Confirm whether the package is owned by the intended npm account or organization.
- If the name is unavailable, open a package naming decision PR. Do not publish renamed packages manually.

### `@anlyx` Scope Permission Denied

- Stop the release.
- Confirm npm organization or scope ownership.
- If access cannot be obtained, follow the `@suhannoh/*` fallback migration plan.

### 2FA or Auth Failure

- Stop and re-authenticate with npm.
- Confirm whether publish requires an OTP or an automation token.
- Do not retry rapidly with partial commands; re-run `npm whoami` first.

### Workspace Dependency Not Resolved

- Stop publishing dependents.
- Inspect packed `package/package.json` from the tarball and confirm no `workspace:*` ranges remain.
- Use pnpm publish/pack behavior that converts workspace ranges, or prepare an explicit dependency range fix PR.

### Packed Package Missing `dist` or `bin`

- Stop release.
- Run `corepack pnpm -r build` and the affected package `pack --dry-run`.
- Fix package `files`, `exports`, build scripts, or CLI `bin` metadata in a PR before publishing.

### Publish Partially Succeeded

- Do not unpublish unless absolutely necessary and allowed by npm policy.
- Record which packages and versions succeeded.
- Fix the failed package in a follow-up patch version if needed.
- Continue only if dependency order and registry state are understood.

### CLI Installs but `npx anlyx` Fails

- Check `npm view anlyx bin`.
- Install in a clean temp directory and inspect `node_modules/.bin/anlyx`.
- Confirm `packages/cli/dist/index.js` has a shebang and is included in the tarball.
- If the issue is package metadata or dist output, publish a patch release after fixing it in Git.
