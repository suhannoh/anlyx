# Anlyx v0.1 Scope Lock

## Purpose

This document fixes the implementation boundary for Anlyx v0.1. It has higher priority than the broader PRD when scope decisions conflict.

Product intent and evidence hierarchy are fixed in [Product Spine](./product-spine.md). Scope decisions MUST preserve the interaction-first rule:

```txt
real app user action -> browser-observed request -> matched scanned backend flow
```

Live browser evidence, scanned backend/frontend evidence, and inferred server-side fetch evidence MUST remain visually and textually distinct.

## v0.1 Core Goal

Anlyx v0.1 MUST make one Spring Boot + Next.js App Router application understandable through:

- Full-page Live Workspace as the primary analysis surface
- Browser request capture from the real local frontend
- Recent events, Summary, Timing, Diagram, and Evidence inspector views for the selected request
- Endpoint Map as fallback/debug visualization
- Page Storyboard as fallback/debug capture evidence
- Main Flow / Sub Flow distinction
- Replay Lite
- Confidence-aware best-effort analysis

v0.1 MUST NOT attempt to become a universal code analysis platform.
v0.1 MUST NOT become a generic network log. Passive implementation traffic MAY be recorded, but it MUST NOT replace the selected main flow for a user's direct action.

## Deep Support

v0.1 Deep Support is limited to:

- Backend: Spring Boot
- Frontend: Next.js App Router

Deep Support means Anlyx may infer Controller, Service, Repository, Database Table, Page, API call, Main Flow, Sub Flow, and Replay Lite relationships.

## Basic Support

v0.1 Basic Support is limited to:

- OpenAPI backend import

OpenAPI Basic Support MAY create endpoints, request schemas, response schemas, and minimal schema nodes. It MUST NOT create internal Controller, Service, Repository, or Database nodes from OpenAPI alone.

## Required v0.1 Features

- Three-command onboarding target: `npm i -D anlyx`, `npx anlyx init`, `npx anlyx dev`
- `npx anlyx init`
- `npx anlyx scan`
- `npx anlyx dev`
- `anlyx.config.ts` generation
- `anlyx dev` as the primary developer command that can prepare analysis data, run the Anlyx runtime, open the real local frontend, and open Anlyx Workspace
- `anlyx dev` Workspace Mode that keeps `frontend.baseUrl` as the real app URL and serves the Anlyx live workspace from the local Anlyx server
- Development-only browser capture runtime for Next.js App Router and manual React SPA setups
- Development-only Next.js server fetch bridge for local Server Component/API fetch observation
- Small capture badge on the real local app when useful, with an entry point to the workspace
- Browser-side `fetch` and `XMLHttpRequest` API event collection during local development
- Next.js server-side `fetch` event collection during local development, clearly labeled separately from browser capture
- Request-to-endpoint matching against scanned `ScanResult.endpoints`
- Full-page Workspace showing Recent events, Summary, Timing, Diagram, confidence, and evidence for the selected request
- Historical viewer routes may remain only as Live Workspace aliases for compatibility
- Spring Boot endpoint discovery
- Spring Boot Controller to Service to Repository best-effort flow inference
- JPA Entity and Repository based DB table inference
- Confidence display for inferred analysis
- OpenAPI endpoint import
- Manual frontend URLs for OpenAPI Basic Support page capture
- Next.js App Router page discovery
- Dynamic route handling through `sampleParams`
- Playwright page capture
- Long page segment capture
- Playwright `storageState` support for authenticated captures
- Network API call collection during capture
- Page to Endpoint linking
- Main Flow and Sub Flow visual distinction
- Local web UI
- Reverse-proxy or large overlay modes only as legacy fallback/debug behavior, not as the default product surface
- Clean Light default product UI
- Dark Replay demo direction

## Explicitly Excluded From v0.1

v0.1 MUST NOT include:

- FastAPI Deep Support
- Express Deep Support
- NestJS Adapter
- Flask or Django Adapter
- React Router Deep Support
- Impact View
- Diff View
- GitHub Actions report generation
- Mermaid export
- PNG or SVG export
- Static HTML report export
- Advanced Replay timeline
- Java Agent runtime tracing
- Production runtime tracing or always-on traffic collection
- Server-side API replay or mutation
- Browser extension distribution
- LLM-generated flow summaries
- Automatic security analysis
- Automatic performance diagnosis
- DB query optimization recommendations
- Full call graph generation
- 3D UI
- Always-on particle, glow, or decorative animation systems

## Deferred Candidates

v0.2 MAY consider:

- FastAPI Enhanced Support
- Express Enhanced Support
- React Router Enhanced Support
- Static report export
- Mermaid export
- PNG or SVG export
- Impact View
- Advanced Replay
- Stronger Cache, External API, and Event nodes
- Prisma, TypeORM, or SQLAlchemy DB connection support
- ELK.js based advanced layout

v0.3 MAY consider:

- Java Agent runtime tracing
- PR Flow Diff
- GitHub Actions report generation
- Adapter SDK
- Plugin system
- LLM-based flow summaries
- Large monorepo support
- Multi-service architecture maps

## Scope Change Rules

- Any scope expansion MUST update this document before implementation.
- Any Data Contract change MUST update `docs/contracts/data-contract.md` and expected fixture JSON.
- Any Adapter behavior change MUST update the relevant adapter rule document.
- Any UI behavior change MUST update the relevant design document.
- Any feature listed as excluded MUST be rejected unless this document is intentionally revised first.

## Agent Guidance

When unsure, choose the smaller v0.1 behavior. Prefer a visible `unknown`, `pending`, or `failed` state over speculative support.
